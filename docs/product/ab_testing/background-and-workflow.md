# A/B 实验背景和操作流程

A/B 实验的目标不是“找一个数据更高的版本”，而是在可控条件下判断某个改动是否带来了可信的因果影响。Firebase、Remote Config、内部实验平台都只是工具，核心流程是一致的。

## 一、实验立项

### 1. 明确问题

实验前先把问题写具体，不要只写“提升指标”。

不清楚的写法：

```text
提升留存。
```

更好的写法：

```text
目标用户在首次体验后的回访动力不足，D1/D7 留存低于预期。
我们希望通过某个产品改动，提高用户再次回来的概率。
```

问题写得越清楚，后续指标、样本量和结论越不容易跑偏。

### 2. 写出可验证假设

一个实验假设至少包含五部分：

| 要素 | 示例 |
| --- | --- |
| 目标人群 | 新用户、老用户、某版本用户、某国家用户 |
| 改动内容 | 参数、入口、流程、文案、策略 |
| 作用机制 | 为什么这个改动会影响用户行为 |
| 主指标 | 最想提升的指标 |
| 风险指标 | 不能被明显伤害的指标 |

推荐模板：

```text
如果对 [目标人群] 做 [改动]，
因为 [作用机制]，
所以 [主指标] 会提升，
同时 [护栏指标] 不应出现明显负向变化。
```

### 3. 定义最小可接受收益

实验前要先定义 MDE，也就是最小可检测效果。它不只是统计概念，也是业务阈值。

例如：

```text
如果主指标提升低于 0.2pp，即使统计显著，也不足以覆盖上线成本。
如果主指标提升达到 0.5pp，且护栏指标稳定，则值得考虑上线。
```

没有 MDE 的实验，容易陷入两种问题：

- 样本量不足，跑完也无法判断。
- 样本量很大，微小差异也显著，但业务上不值得做。

## 二、实验设计

### 1. 选择实验单位

实验单位是随机分组的最小对象。

| 实验单位 | 适用场景 | 风险 |
| --- | --- | --- |
| 用户级 | 多数产品实验 | 需要稳定用户 ID |
| 设备级 | 未登录场景 | 多设备用户可能跨组 |
| 会话级 | 短期界面实验 | 同一用户可能看到多个版本 |
| 地区/门店/服务器级 | 强网络效应场景 | 样本少，方差大 |

多数 App 实验优先使用用户级分流。用户一旦进入某组，应尽量保持稳定，避免重复切换。

### 2. 定义目标人群

目标人群决定实验结论能外推到哪里。常见维度包括：

- 新用户或老用户。
- 平台。
- App 版本。
- 国家或地区。
- 渠道。
- 是否满足某个前置行为。

如果实验只覆盖 Android 新用户，结论就不要直接说“对全量用户有效”。

### 3. 设计分组

最基础的是 A/B 两组：

```text
A 组：baseline，对照组，保持原方案
B 组：variant，实验组，使用新方案
```

多组实验可以比较多个方案，但会摊薄每组样本。每多一个 variant，单组样本量都会下降，实验周期通常会变长。

分组原则：

- 对照组必须稳定代表旧方案。
- 实验组之间的差异要可解释。
- 如果一次改多个变量，要明确这是组合方案实验，不要强行归因到单个变量。

### 4. 选择指标

指标分三类。

| 指标类型 | 作用 | 示例 |
| --- | --- | --- |
| 主指标 | 决定实验是否成功 | 留存率、转化率、收入、完成率 |
| 护栏指标 | 防止局部收益伤害整体 | 崩溃率、卸载率、投诉率、长期留存 |
| 诊断指标 | 解释为什么变化 | 曝光、点击、漏斗分步、行为次数 |

主指标最好只有一个。指标越多，越容易在随机波动里找到“看起来显著”的结果。

### 5. 设置实验周期

实验周期由三件事决定：

- 每天进入实验的人数。
- 每组所需样本量。
- 指标成熟窗口。

例如要看 D7 留存，不能只跑 7 天。最后一天进入实验的用户还需要再等 7 天，D7 指标才成熟。

## 三、样本量估算

样本量估算应该在实验启动前完成。它回答的问题是：

> 我们至少需要多少用户，才能比较有把握地检测到自己关心的变化？

### 1. 需要先确定四个输入

| 输入 | 含义 |
| --- | --- |
| 基线值 | 当前对照组预期指标，例如 D1 留存 40% |
| MDE | 希望识别的最小变化，例如 +2pp |
| 显著性水平 | 常用 0.05，控制假阳性 |
| 检验功效 | 常用 80% 或 90%，控制漏检 |

### 2. 粗略公式直觉

比例类指标每组样本量近似为：

$$
n \approx \frac{(Z_{\alpha/2}+Z_{\beta})^2[p_1(1-p_1)+p_2(1-p_2)]}{(p_2-p_1)^2}
$$

均值类指标每组样本量近似为：

$$
n \approx \frac{2(Z_{\alpha/2}+Z_{\beta})^2\sigma^2}{\delta^2}
$$

这里不需要死记公式，记住三个直觉就够：

- 想识别越小的提升，需要越多样本。
- 指标波动越大，需要越多样本。
- 组数越多，每组样本越少，实验越难显著。

### 3. 实验时长估算

$$
D_{\mathrm{entry}} = \left\lceil \frac{n \times k}{d} \right\rceil
$$

其中，\(D_{\mathrm{entry}}\) 表示进组天数，\(n\) 表示每组所需样本量，\(k\) 表示实验组数，\(d\) 表示日均进入实验用户数。

如果还要等待 D7 指标成熟：

$$
T_{\mathrm{total}} = D_{\mathrm{entry}} + 7
$$

其中，\(T_{\mathrm{total}}\) 表示从实验开始到 D7 指标成熟的总等待时间。

实验开始前就应该知道大概需要跑多久。否则中途看到结果波动，很容易主观决定“继续跑”或“提前停”。

## 四、上线前 QA

上线前检查四类问题。

### 1. 分流逻辑

- 用户是否只进入一个组。
- 同一个用户是否保持组别稳定。
- 目标人群是否正确。
- 流量比例是否符合设计。

### 2. 参数或功能生效

如果使用 Firebase Remote Config，要确认：

- baseline 和 variant 参数值正确。
- 客户端默认值可兜底。
- fetch / activate 时机符合预期。
- 老版本不会因为新参数出错。

### 3. 埋点

- 主指标所需事件是否完整。
- 诊断指标是否覆盖关键漏斗。
- 激活事件是否发生在用户受到实验影响之前。
- 对照组和实验组埋点口径是否一致。

### 4. 风险监控

- 崩溃、报错、卸载、投诉等指标是否能监控。
- 是否有提前停止实验的风险阈值。
- 是否有回滚方案。

## 五、实验运行后的分流验证

实验开始后，第一件事不是看主指标，而是验证分流是否可信。

### 1. 样本比例校验

如果设计是 50/50 分流，实际结果长期偏成 55/45 或 60/40，就需要排查。常见方法是 SRM 检验。

SRM 的核心公式：

$$
\chi^2 = \sum \frac{(O_i - E_i)^2}{E_i}
$$

其中：

- \(O_i\)：第 \(i\) 组实际人数。
- \(E_i\)：第 \(i\) 组期望人数。

两组 50/50 时，自由度为 1：

| 卡方统计量 | 大致含义 | 建议 |
| --- | --- | --- |
| 小于 3.841 | 未发现明显比例异常 | 继续观察 |
| 3.841 到 6.635 | 有异常信号 | 排查分流和埋点 |
| 大于 6.635 | 明显 SRM 风险 | 不建议直接使用实验结论 |

SRM 不是证明实验有效，而是判断实验有没有失去随机分流基础。

### 2. 用户属性均衡校验

样本比例正常，不代表实验一定可信。还要看关键属性是否均衡：

- 平台分布。
- 国家分布。
- 版本分布。
- 新老用户比例。
- 渠道分布。
- 实验前历史行为。

如果某组高价值用户明显更多，后续收入差异就可能不是实验导致的。

### 3. 激活漏斗校验

一条完整实验链路通常是：

```text
符合目标人群
-> 被分流
-> 参数或功能生效
-> 触发激活事件
-> 产生目标行为
```

需要检查每一步是否有异常掉量。如果实验组因为技术问题触发激活事件更少，结果会出现样本选择偏差。

## 六、数据成熟与正式分析

正式分析前，要先确认关键指标成熟。

| 指标 | 成熟条件 |
| --- | --- |
| D1 留存 | 进组用户至少过完 1 天 |
| D7 留存 | 进组用户至少过完 7 天 |
| D7 LTV | 进组用户至少有完整 7 天收入窗口 |
| 周期行为 | 至少覆盖工作日和周末 |

正式分析建议按这个顺序：

1. 先确认实验数据质量。
2. 再看主指标差异。
3. 再看护栏指标。
4. 最后用诊断指标解释变化原因。

## 七、决策与复盘

实验结束不等于“显著就上线”。建议按下面问题复盘：

- 实验假设是否被支持。
- 主指标提升幅度是否达到 MDE。
- 置信区间下界是否仍可接受。
- 护栏指标是否稳定。
- 结果是否在关键人群和 cohort 中一致。
- 是否存在分流、埋点、样本量或成熟窗口问题。
- 下一步是上线、放量、重做、继续观察还是放弃。

一个好的实验流程，核心是让每一步都提前定义。不要等看到结果后，再决定用哪个口径、看哪个指标、跑到哪天结束。
